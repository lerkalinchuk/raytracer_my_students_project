# PyRayTrace – трассировка лучей на Python

## Идея проекта

Программа строит 3D-изображение методом трассировки лучей. Для каждого пикселя экрана пускается луч из камеры, вычисляется его пересечение с объектами сцены, и на основе модели освещения определяется цвет пикселя. Все вычисления ускоряются с помощью Numba JIT-компиляции. Проект демонстрирует применение векторной алгебры, решения квадратных уравнений и моделирования распространения света в компьютерной графике.

## Модель освещения

Используется модель Phong (1975) с рекурсивными отражениями Whitted (1980). Формула цвета пикселя:

Цвет = Ambient + Diffuse + Specular + Reflection

- **Ambient** — фоновое (окружающее) освещение. Моделирует равномерное освещение от непрямого света, который многократно отразился от других поверхностей в сцене. Создаёт базовый уровень освещённости даже в тени. Зависит от коэффициента фонового отражения материала (свойство поверхности) и интенсивности глобального фонового освещения (свойство сцены).

- **Diffuse** — диффузное (рассеянное) отражение. Моделирует рассеяние света во всех направлениях от шероховатой поверхности (например, бумага, матовая краска). Яркость зависит от угла падения света по закону Ламберта. Физический смысл: чем более перпендикулярно свет падает на поверхность, тем ярче она выглядит.

- **Specular** — зеркальное (спекулярное) отражение. Моделирует блики от гладких/полированных поверхностей (металл, стекло, пластик). Зависит от положения наблюдателя.

- **Reflection** — вклад рекурсивных отражений (метод Уиттеда). Моделирует реалистичные зеркальные отражения других объектов сцены. Позволяет создавать эффекты «зеркала в зеркале», отражения бликов и т.д.

## План фич по итерациям

### Итерация 1 (26 апреля) – полная базовая трассировка

На первой итерации будет реализован полностью рабочий рейтрейсер с базовым набором возможностей. Это основа, на которой уже можно рендерить простые сцены с тенями и отражениями.
Будет реализован абстрактный класс, от которого наследуются все геометрические примитивы. На данный момент будут реализованы сфера и плоскость как основные объекты, а цилиндр и эллипсоид будут добавлены позже при необходимости. Каждый объект хранит свой материал и реализует метод hit(), который проверяет пересечение луча с объектом и возвращает точку пересечения, нормаль и расстояние.
Будет реализован класс Camera, который находится в заданной точке пространства и смотрит в заданном направлении. Метод generate_ray(x, y, width, height) для каждого пикселя создаёт луч, проходящий через этот пиксель с учётом перспективной проекции. Камера также поддерживает поворот вокруг вертикальной оси для возможности смены ракурса.
Будет реализована полная модель освещения Phong, включающая все четыре компоненты: Ambient (фоновый свет), Diffuse (рассеянный свет, зависящий от угла падения по закону Ламберта), Specular (блики на блестящих поверхностях) и Reflection (рекурсивные отражения от зеркальных поверхностей с глубиной до 3-4 уровней). Также будут реализованы тени через shadow rays: из точки пересечения пускается луч к источнику света, и если на пути встречается другой объект, точка считается находящейся в тени, и diffuse и specular компоненты обнуляются.
Будет реализовано ускорение с помощью Numba. Функции trace_ray() и render() будут обёрнуты в декоратор @njit(parallel=True), что позволит распараллелить основной цикл рендеринга по строкам изображения. Ожидаемое ускорение составит 50-100 раз по сравнению с чистым Python.
Будет реализована простая тестовая сцена, содержащая три сферы разного цвета (красную, зелёную, синюю), расположенные на разном расстоянии от камеры, плоскость-пол под ними и один источник белого света. Это позволит проверить работоспособность всех компонентов: тени от сфер на пол, блики на блестящих поверхностях (если материал металлический), отражения если добавить зеркальную сферу.
Будет написан main.py, который загружает тестовую сцену, создаёт камеру в точке (0, 1, 0) направленную на центр сцены (0, 0, -5), запускает рендер с разрешением 800×600 и сохраняет результат в файл output.png. При желании можно добавить аргументы командной строки для выбора сцены, разрешения и количества сэмплов.

### Итерация 2 (10 мая) – сложные сцены и анимация

На второй итерации основное внимание будет уделено созданию сложных визуально привлекательных сцен и анимации для устной сдачи. Базовая математика и алгоритмы уже готовы с первой итерации, поэтому на этом этапе нужно будет добавлять новые объекты и сцены, а также улучшать качество рендеринга.

Будут добавлены дополнительные геометрические примитивы: цилиндр (понадобится для рук и ног персонажа) и эллипсоид (для тела персонажа). Они будут наследоваться от Hittable и реализовывать метод hit(). Для цилиндра потребуется учитывать ограничение по высоте, для эллипсоида – использовать преобразование координат к единичной сфере.

Будет создана сложная сцена с персонажем. Персонаж будет собран из примитивов: голова – сфера, тело – эллипсоид или параллелепипед, руки и ноги – цилиндры, глаза и нос – маленькие сферы. Для разных частей тела будут использованы разные материалы: кожа для головы и рук, ткань для одежды, пластик для глаз. 
Будет улучшено качество рендеринга. Будет реализована многократная выборка (multisampling): для каждого пикселя будет пускаться не один луч, а 256-1024 лучей со случайным смещением внутри пикселя, после чего результаты усредняются. Это даст сглаживание и уберёт шум. Также будет реализован area light (прямоугольный источник света) для получения мягких теней. Вместо точечного источника будет создан прямоугольник, испускающий свет, и для проверки тени будет пускаться несколько лучей к разным точкам этого прямоугольника.


Будет создана анимация вращения камеры. Для этого будет написан скрипт, который рендерит последовательность кадров (например, 36 кадров при шаге 10 градусов или 72 кадра при шаге 5 градусов). Каждый кадр рендерится с достаточно высоким качеством (256 сэмплов на пиксель) и сохраняется в отдельный файл. Затем с помощью библиотеки imageio или утилиты ffmpeg кадры собираются в видео (GIF или MP4). Это видео будет показано на устной сдаче, чтобы продемонстрировать возможности проекта в динамике.
